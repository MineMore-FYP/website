Results were evaluated under two main criteria; Time and Flexibility.

Time

Channel coordination slightly increases the time taken to complete, while using Parsl reduces the time. There is a communication delay when employing all the machines in the cluster with our framework. We think this may become negligible as the dataset grows larger.

The time taken to complete could be ranked as below; (from best to worst)

Rank Go Parsl
1 Without With
2 With With
3 Without Without
4 With Without

Using cores of a local machine significantly reduces the time consumed. The time taken, utilizing our cluster is similar to this value, owing to the cluster machines not being adequately strong and the communication delay involved.

Flexibility

Without Channel Coordination :

The individual modules need to be executed from a shell script.

It is not possible to perform dynamic decision making at run-time.

Less time taken to complete.

Without Implicit Parallelism :

Cannot run on different configurations (threads/ cores/ clusters) easily. Would require MPI programming or a similar technology.

Longer time taken to complete.

SciFlow provides benefits along the dimensions of both time and flexibility, as measured on this case study. Using SciFlow on a Data Analytics workflow defenielty enhances the user’s expressiveness, and allows them to define a dynamic workflow. Channel coordination provided control to an event-driven workflow. Control is separated from coordination, allowing Data Analytics modules to be reusable, as is the general practice in the domain.
